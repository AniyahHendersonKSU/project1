Topic 02

CSS Analysis: Selectors, Layout & Effects

After researching I see that none of the template's visual design comes from a hand-written CSS selector instead it comes from applying W3.CSS's pre-built utility classes directly in the HTML. Only nine lines of custom CSS appear in the whole document. That single fact is the most important thing to understand about how this template works.

1. Utility-class selectors vs. hand-written selectors

A typical hand-rolled stylesheet selects elements by tag, class, ID, or combinations of these, then groups related declarations together one rule for "card," one rule for "button," etc. W3.CSS inverts this: instead of writing .card { border: 1px solid #ccc; padding: 16px; } once, you compose several tiny classes on the element itself: class="w3-card w3-padding w3-white". The CSS selectors themselves (inside w3.css, which the template only links to, never edits) are simple single-class selectors like .w3-card, .w3-padding, each responsible for exactly one property.

This is the "utility-first" approach to CSS. It trades stylesheet readability for HTML predictability but you can guess what an element looks like just by reading its class list, without ever opening the CSS file. The cost shows up in the markup: that one <div> above carries four classes to do a job that a single custom class name could describe more clearly.

2. The custom CSS that is written, selector by selector

SelectorTypeWhat it does
body, htmlType (grouped)Forces full page height and sets the base font
.bgimgClassPositions and sizes the header's background photo
.menuClassHides both tab panels by default (display: none)

That's the entire custom stylesheet. Just three selectors. Everything else (colors, spacing, the responsive column grid, hover states) is inherited from w3.css. This means the "layout design" of the page is really a design decision made once, by the framework's authors, and reused here rather than authored fresh.

3. Layout system: a 12-column float/inline-block grid

W3.CSS's grid predates modern CSS Grid and Flexbox. The nav bar uses w3-row + four w3-col s3 children — "s3" meaning 3 of 12 columns on small screens, so four equal columns exactly fill the row. It's effective and genuinely responsive (a one-line media-query swap collapses columns on narrow viewports), but it depends on precise column-count math rather than the more flexible flex: 1 or grid-template-columns: repeat(...) approaches available in current CSS. This site you're reading now uses real CSS Grid and Flexbox instead, partly to make that contrast visible.

4. Special effects: grayscale, display toggling, and image tricks

  • w3-grayscale / w3-grayscale-min: applies a CSS filter: grayscale() to the header photo and the page background, then relies on :hover or JS-free defaults to control intensity — a good example of a pure-CSS visual effect with zero JavaScript.
  • w3-display-* classes: position the "Open 6am–5pm" tag, the centered "the Cafe" title, and the address, all inside one position: relative container using position: absolute variants — a CSS layout technique (absolute positioning inside a relative parent) rather than a framework invention, just packaged into reusable classes.
  • The Eat/Drink toggle is CSS's display: none driven entirely by JavaScript class swapping, not a CSS-only solution (no :target or checkbox hack) — meaning the tabs won't work at all if JavaScript fails to load.

This image sits inside the Menu section using nothing but style="width:100%;max-width:1000px" written inline which is another example of styling done in the HTML itself rather than through a CSS selector, which ties back to the specificity point below.

5. Specificity: mostly flat, one landmine

Because nearly every rule is a single-class selector at the same specificity level, there's little fighting between rules, a real benefit of the utility-class approach. The one place this breaks is the inline style="font-size:90px" on the "the Cafe" heading span: inline styles have higher specificity than any class, so if you wanted to override that font size from the stylesheet later, you'd have to fight the HTML itself, not just add a more specific selector.

Takeaway: a framework can make a page look finished with almost no custom CSS, but understanding what a class actually does still requires reading the framework's source, not just the page's.
